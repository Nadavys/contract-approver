# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run dev` — start the Vite dev server
- `npm run build` — type-check (`tsc -b`) then production build (`vite build`)
- `npm run lint` — run oxlint (see `.oxlintrc.json`)
- `npm run preview` — preview the production build
- `npm run storybook` — start Storybook on port 6006
- `npm run build-storybook` — build the static Storybook site

There is no test suite. Verification loop for any change is: `npx tsc -b` and `npx oxlint`, both must produce no output. `tsconfig.app.json` has `noUnusedLocals`/`noUnusedParameters` on, so unused imports (e.g. React 19's JSX runtime doesn't need `import React`) fail the build.

To check a single file compiles/serves without a full type-check, run `npm run dev` in the background and `curl` the file's dev-server URL, then kill the server.

## Architecture

This is **not** a general dashboard template — it is a purpose-built **PDF document review tool** with two real screens: a Documents list (landing page) and a PDF review workflow (view, search, resolve issues, approve, see a confirmation). It started from a generic MUI dashboard scaffold; the unused leftover pages/nav-items file from that scaffold have since been removed as dead code.

### Routing (`react-router-dom`)

`src/main.tsx` wraps `<App />` in a `BrowserRouter`. `App.tsx` fetches `useDocumentStore`'s data once on mount (see State below) and only renders `<Routes>` once that's `status === 'loaded'` — otherwise it shows a full-page spinner (or an error message) in place of the routes, so nothing downstream ever has to handle an empty `reviews` record. Once loaded: `"/"` → `DocumentsPage`, `"/documents/:id"` → a `DocumentRoute` guard that renders `ReviewScreen` (the review/approved flow) only if `:id` exists in `useDocumentStore`'s `reviews` record, else redirects to `"/"`. `src/routes.ts` exports `documentPath(id)` and `useDocumentId()` — a hook that reads `:id` off `useParams()`, falling back to `Object.keys(reviews)[0]` (read live off the store, not a frozen constant — `reviews` isn't known at module-import time now that it's fetched) — so any component under the `/documents/:id` route can look up "the current document" without prop-drilling the id. Any story or test that renders `<App />` needs a `MemoryRouter` wrapper (see `App.stories.tsx` / `App.documentsList.stories.tsx`), since `App`/`DocumentsPage`/`ApprovedPage` all use `Routes`/`useParams`/`Link` internally.

### State (Zustand, no Context/Redux)

Components read directly from stores rather than receiving props — this is the established pattern here, not prop-drilling:
- `src/store/useUserStore.ts` — current user (hardcoded).
- `src/store/useThemeStore.ts` — `{ mode: 'light' | 'dark', toggleMode }`, drives `App.tsx`'s `ThemeProvider`.
- `src/store/useDocumentStore.ts` — `reviews: Record<string, Review>` (see `src/types/review.ts` for `Review`/`Issue`/`ReviewUser`/`ReviewDocument` shapes), plus `status: 'idle' | 'loading' | 'loaded' | 'error'` and `error: string | null`. `reviews` starts empty; `loadReviews()` fetches `public/mock_data/documents_api_mock.json` (a stand-in for a real `GET /api/documents` response — an array of documents) and reduces it into the `reviews` record via `toReview()`, which stamps `resolved: false` onto every issue (the source data doesn't carry that flag — it's app-local review state). `loadReviews()` is guarded against re-entry (a no-op if `status` is already `'loading'`/`'loaded'`), which is what lets `App.tsx` call it unconditionally on every mount without worrying about re-fetching or racing a story's own seeded state. `ReviewJson` (the raw fetch-response shape) and `toReview()` are exported so Storybook stories can seed `reviews` directly from the same fixture file — without going through `loadReviews()` — and mark `status: 'loaded'` themselves; otherwise `App`'s mount effect would kick off a real fetch that could resolve after a story's `beforeEach` and clobber its custom state (see the top of `App.stories.tsx`). Components look up `state.reviews[docId]` via `useDocumentId()` rather than reading a single `review` field. Also owns `toggleResolved(docId, issueId)` and `approve(docId)`, plus derived selectors (`selectOpenCount`, `selectCanApprove`, `selectBlockingRemaining`, `selectBlockingProgress`, `selectMinorCount`) — these are plain functions that take a single `Review` and are computed from `review.issues` on every read, not stored state, so they can never drift out of sync with the `resolved` flags. Follow this pattern for new derived values instead of caching them in the store. `approve(docId)` re-checks `selectCanApprove` itself (not just the UI's disabled state) before flipping that review's `status` to `'approved'` and stamping `approved_at`. Note: the Storybook story files still `import` this same JSON directly (as a static fixture, not via `fetch`) — Vite/Vitest print a "assets in public directory cannot be imported from JavaScript" warning for that during `test-storybook` (Vitest's browser-mode dev server disallows it); non-fatal, the data loads correctly, but it's an acknowledged deviation from Vite's convention. `public/mock_data/review_mock.json` (the single-object precursor to `documents_api_mock.json`) has been removed — `documents_api_mock.json` is the only mock data file now.
- `src/store/useViewerStore.ts` — `{ jumpPage, nonce, requestJump(page) }`, a small pub/sub used only for the issue→page-jump interaction (see below); `nonce` increments on every call so repeat clicks on the same page still retrigger the effect.
- `src/store/documentRows.ts` — not a store itself, but owns the `DocumentRow`/`IssueSummary` view-model types and `selectDocumentRows(reviews)`, a pure derivation of the documents table's rows straight from `useDocumentStore`'s `reviews`. There's no separate mock-rows store: the table and the review-screen sidebar read the same underlying data, so they can't drift out of sync (mirrors the derived-selector pattern above).

All app data lives in these stores — there is no real backend. `useDocumentStore`'s data is fetched (from a static JSON file standing in for an API response) rather than hardcoded, so it has genuine async loading/error states; everything else is still hardcoded in-store.

### Theming — "Slate" design system

The app follows the **Slate** visual design (the original design-handoff folder this was implemented from is no longer part of the repo). `src/theme.ts` exports two things:
- `getTheme(mode: PaletteMode)` — a factory (not a static theme), so light/dark mode can be switched at runtime. `App.tsx` memoizes `getTheme(mode)` off `useThemeStore`. Palette: violet accent `#7C3AED` as `primary`, `error`/`warning`/`success` mapped to the Slate critical/major/success colors, `background.default = #F2F3F7` in light mode, `typography.fontFamily` = Manrope. **Only override `palette.text`/`palette.divider` for light mode** — setting them to `undefined` for dark mode (instead of omitting the keys) breaks MUI's palette merge and crashes `CssBaseline` (`Cannot read properties of undefined (reading 'primary')`); spread the override in conditionally instead.
- `slate` — a plain token object (colors, font stacks, `severityColor` map, `approveShadow`) for the soft tints/borders/severity colors that don't map onto MUI's standard palette roles (e.g. `slate.critical.softBg`, `slate.chip.bg`). Components import this directly and use it in `sx` alongside theme-driven MUI props.

Fonts (Manrope, Space Grotesk, Source Serif 4) are loaded via Google Fonts `<link>` in `index.html`, not npm packages.

Component style overrides (`MuiCard`, `MuiAppBar`, `MuiDrawer`) use `theme.palette.divider` so borders adapt to mode automatically — follow this pattern for new overrides instead of hardcoding colors.

Storybook's `.storybook/preview.tsx` wraps every story in the same `getTheme`/`ThemeProvider`/`CssBaseline`, with a toolbar toggle (`globalTypes.theme`) to preview light/dark — component stories should rely on this decorator rather than wrapping themselves in their own `ThemeProvider`.

### PDF viewer and search (`src/components/documents/`)

- `usePdfSearch.ts` is the shared hook (returned type `PdfSearch`) owning all search state — query, match count/index, refs, and the `customTextRenderer` passed to react-pdf's `Page` to inject `<mark class="pdf-search-mark">` into matched text. It is instantiated once in `ReviewScreen.tsx` and passed down to both `PdfSearchBar` (header UI) and `PdfViewerPage` (rendering + the scoped container ref).
- `PdfViewerPage.tsx` reads the file it renders from `state.reviews[docId].document.pdf_url` (via `useDocumentStore`/`useDocumentId()`), not a hardcoded constant — react-pdf's `<Document file={...}>` always points at whichever document the current route resolves to.
- Match highlighting must stay visible over the PDF canvas: use plain `rgba()` backgrounds in `pdfSearch.css`, not `mix-blend-mode` — the text layer's `position: absolute` + `z-index` creates an isolated stacking context that blend-mode cannot see through to the canvas.
- Because react-pdf renders each page's text layer asynchronously (and match `<mark>`s land at different times per page), match counting is driven by a `MutationObserver` on the search container rather than a fixed timer/`requestAnimationFrame` delay — do not revert to timer-based counting.
- Centering/stacking the pages must be applied to react-pdf's own `Document` wrapper via its `className` prop (`.pdf-document-stack`), not the outer `Card` — `Document` renders its own wrapper div, so flex layout on an ancestor of it has no effect on how pages stack.
- Cmd+F/Ctrl+F is intercepted at the `window` level and scoped to just the PDF (not the sidebar/header) by only searching within `usePdfSearch`'s `containerRef`.
- **Issue → page jump**: clicking an issue card calls `useViewerStore.requestJump(page)`. `PdfViewerPage` reads `nonce` (not `jumpPage`) as its effect dependency and pulls the current `jumpPage` from `useViewerStore.getState()` inside the effect — this keeps the effect's dep list to just `nonce`/`containerRef` while always acting on the latest requested page. Each PDF page wrapper has a `pdf-page-<n>` DOM id; the jump scrolls the search `containerRef` to that element's offset (not `scrollIntoView`, to avoid disrupting the scroll container) and toggles the `.pdf-page-pulse` CSS class (keyframe in `pdfSearch.css`) to flash it.

### Layout shell (`src/components/layout/`)

`SideMenu.tsx` is a fixed 308px drawer, top to bottom: brand row (`public/logo.svg`) → document metadata (`DocumentInfo`, reading `useDocumentStore`) → issues header (open/total count) → issues list (`IssuesList`, scrollable, clickable cards with a resolve toggle) → approve gate (`ApproveGate`, inline in `SideMenu.tsx`). The gate has two mutually-exclusive states driven by `selectCanApprove`: a rose "blocked" card with a per-blocking-issue progress bar + disabled button, or a green "clear" card + enabled violet button (labeled "Approve this contract") wired to `useDocumentStore.approve()`. Minor issues never block approval. There's no user profile in the drawer — that's on the right side of the header instead (see `UserChip.tsx` below), freeing up vertical space in the drawer. `AppNavbar.tsx` takes composable `children` (used for `PdfSearchBar`) rather than a fixed title, plus a light/dark toggle and a notifications bell (both styled as outline icon buttons), a "⌘F to search" hint pill, and `UserChip`. The bell opens `WIPDialog.tsx`, a generic "not implemented" modal (there is no real settings page).
- `UserChip.tsx` — avatar + name/email, reading `useUserStore` directly. Shared between `AppNavbar.tsx` (review screen header) and `DocumentsPage.tsx` (documents list top bar) — the one place per page the user's identity shows, now on the right side of the header rather than pinned to the bottom of each page's own left panel (drawer / nav rail). Takes `nameColor`/`emailColor` as props rather than picking its own, because the two hosts need different color sources: `AppNavbar`'s `AppBar` background is mode-aware, so it passes theme-path colors (`'text.primary'`/`'text.secondary'`); `DocumentsPage`'s top bar is still styled with static `slate.text.*` hex throughout (that page's dark-mode support is incomplete — untested by any story combination — so it stays consistent with the rest of that file rather than mixing static and theme-aware colors on the same bar). Not shown on the approved-confirmation screen, since that screen renders no header at all (see Approved flow below).

### Approved flow (post-approval confirmation)

`Review.status` (`src/types/review.ts`) is exactly `'created' | 'processing' | 'on_review' | 'approved'` — no other values exist anywhere in the app; don't widen it for illustrative/mock purposes. Once a given document's `status === 'approved'`, several components switch to a read-only/approved variant for that document, all keyed off that one field (no separate route/page component drives this — see `ReviewScreen.tsx`'s conditional render):
- `DocumentInfo.tsx` — status pill gets a checkmark and turns green; the meta line reads "Approved {date}" off `approved_at` instead of "Uploaded {date}" off `uploaded_at`.
- `IssuesList.tsx` — cards become non-interactive (`readOnly` prop threaded from `status`): no `onClick` jump, no resolve toggle, dimmed `#FAFAFC` bg. The action row shows a static chip instead of a button — **by severity, not by the `resolved` flag**: any non-minor (critical/major) issue always reads "✓ Resolved" (approval is gated on all of them being resolved, so this is guaranteed true), and every minor issue always reads "Noted" regardless of whether it was individually toggled. Don't wire this off `issue.resolved` for minors — the design intentionally shows "Noted" either way.
- `SideMenu.tsx` — issues header reads "{total} total · locked" instead of "{open} open · {total} total". The bottom slot is one of three mutually-exclusive things, keyed off `status`: `ApproveGate` (approve button + gate card) only when `status === 'on_review'`; `ApprovedStamp` (a green "Contract Approved" receipt-style card with no button) when `status === 'approved' && approved_at` is set; otherwise **nothing** — a `created`/`processing` document has no actionable approval UI to show yet, so the gate is hidden entirely rather than rendering a disabled button. (If `approved_at` is missing despite `status === 'approved'` — e.g. set directly via Storybook controls rather than through a real `approve()` call — it also renders nothing, rather than crashing.)
- `ApprovedPage.tsx` — the confirmation main-area content (success mark, headline "Contract approved by {user}", receipt card with resolved/flagged counts pulled from `selectBlockingProgress`/`selectMinorCount`, 2-node "Reviewed → Approved" progress tracker — both nodes complete, since approval is terminal now rather than a hand-off to a separate approver — "View document"/"Back to documents" actions). `ReviewScreen.tsx` renders this **in place of the entire right column** (no `AppNavbar`/search bar) when the current document's `status === 'approved'` and its local `showConfirmation` state is true; that state resets off `[docId, status]` (via a `useEffect`) so both approving and navigating between documents land on the right screen. The two actions differ: "View document" (`onViewDocument` prop) dismisses the confirmation locally to reveal the (now read-only) PDF view, staying on the same route; "Back to documents" is a real `Link` to `"/"` (DocumentsPage).

### Documents list (`DocumentsPage.tsx`) and `StatusPill.tsx`

The landing page (`"/"`) — different chrome from the review screen: a 236px nav rail (`#FBFBFD` bg, brand row, Documents/Approvals/Activity nav items where only "Documents" is the current page and "Settings" opens the shared `WIPDialog`) instead of `SideMenu`, plus a top bar (title, a real name-filter search field, dark/light toggle + notifications bell matching `AppNavbar`'s styling, an "Upload" button that also opens `WIPDialog`, and `UserChip`) and a main area with a stat strip, filter chips, and a documents table.
- **`StatusPill.tsx`** maps every `ReviewStatus` to its label/color/dot-or-check styling in one place; used by both `DocumentInfo.tsx` (sidebar) and `DocumentsPage.tsx` (table) so status treatment stays consistent. The terminal state (`approved`) gets a check icon instead of a dot.
- **Row data**: every row is real, derived via `selectDocumentRows(reviews)` (`src/store/documentRows.ts`) from `useDocumentStore`'s full `reviews` record — table and sidebar share one source of truth. Each row's `Link` points at its own `documentPath(row.id)`. Currently there's only one seeded document, so the table has exactly one row; the derivation is written to map over however many documents `reviews` holds, not to assume a fixed count.
- **Stat strip + filter chips** cover all 3 status buckets a real `Review` can reach: `needs_review` (`created`/`processing`), `in_review` (`on_review`), `approved`. Filter chips and the search field are both functional — they filter the same `visibleRows` list together.

## Notable stack choices (from `notes.rm`)

- Zustand over Redux/Context: simple state needs, no justification for more.
- MUI over Tailwind: "ms-office"-style app benefits from MUI's consistent, mature component set out of the box.
- npm over pnpm: wider community support, despite pnpm being preferred generally.
- No Next.js: this is a simple frontend; an SSR framework isn't justified. Client-side routing (`react-router-dom`) was added once the app grew a second real screen (Documents list); this doesn't change the SSR calculus.
